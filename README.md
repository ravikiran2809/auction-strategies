# IPL Auction Strategy Platform

A research and live-advisory platform for IPL fantasy auction bidding. Combines Monte Carlo simulation, evolutionary parameter optimisation, and an in-browser advisor that delivers real-time recommendations backed by live Python strategies.

---

## Architecture

```
main.py                         CLI entry-point (build-pool, simulate, evolve, serve)
engine/
  pool.py                       Build player pool from ipl_player_stats.csv
  strategies.py                 Manager base class + 19 static strategies + STRATEGY_REGISTRY
  pta_strategies.py             PTA strategies: SquadCompletionBidder, AdaptiveRecoveryManager
  market.py                     MarketAnalyzer, SquadBuilder — fully decoupled, no engine imports
  auction.py                    run_auction(), build_state(), on_sale/on_unsold hooks
  simulation.py                 Monte Carlo engine
  export.py                     pool/insights serialisation
  overrides.py                  Manual projected_points overrides
server.py                       FastAPI server — advisor endpoints
ipl_live_advisor.html           Single-file in-browser advisor (served at /)
player_master.json              179 players: id, name (auction), csv_name, ipl_team, role, auction_set
player_pool.json                135 active pool players (generated by build-pool)
evolved_params.json             CMA-ES evolved params for all 21 strategies
insights.json                   Latest MC simulation results
ipl_2025_schedule.json          74 IPL 2025 matches {match_id, team1, team2}
name_aliases.json               169 auction_name → csv_name mappings
scoring_rules.json              Scoring weights for batting/bowling stats
```

## CLI Commands

```bash
# Build player pool
python main.py build-pool
python main.py build-pool --model weighted --seasons 2022:0.3,2023:0.6,2024:1.0
python main.py build-pool --model recent --last-n 10

# Monte Carlo simulation (6 random strategies per auction)
python main.py simulate --mc 500
python main.py simulate --mc 200 --strategies SquadCompletionBidder AdaptiveRecoveryManager

# Evolve strategy params (CMA-ES recommended)
python main.py evolve --strategy SquadCompletionBidder --algo cmaes --generations 25 --mc-runs 25

# Evolve all strategies against each other (co-evolution)
python main.py evolve --strategy SquadCompletionBidder --algo both --generations 15 --mc-runs 15

# Run a single simulated auction (for inspection)
python main.py run
```

## Strategy Registry (21 strategies)

| Strategy | Type | Co-evolved win share | Win rate/game |
|---|---|---|---|
| AdaptiveRecoveryManager | PTA dynamic | 23.8% | 73.5% |
| SquadCompletionBidder | PTA dynamic | 16.4% | 63.1% |
| FieldAwareAdaptor | Static | 16.4% | 51.9% |
| NominationGambler | Static | 14.6% | 53.3% |
| … 17 others | Static | varies | varies |

**PTA combined: 40.2% win share** — more than 2× the 16.7% equal-share baseline.

### SquadCompletionBidder

Bids based on `target_priority_score` from `SquadBuilder` — a 0–1 score derived from role need, number of alternatives, and cliff size. Key behaviors:

- `priority ≥ 0.80` → must-bid (2.0× WTP multiplier)
- `priority ≥ 0.50` → useful (1.1×)
- `priority < 0.50` → floor bid (0.35×)
- Role cliff boost: if the player is significantly better than the next-best in their role, upgrades the multiplier
- Opponent shading: if must-bid player has high opponent WTP, raises bid to just beat them; shades down on comfortable wins
- Opponent denial: bids harder when opponents are under high role pressure (uniquely possible because SCB tracks opponent rosters)
- Market mode dampening: only applies to sub-must-bid players

### AdaptiveRecoveryManager

Plan-Track-Adapt strategy with proactive normal mode and aggressive recovery:

- **Normal mode**: proactive (2.2× for must-bid), market dampening only on low-priority players
- **Recovery trigger** (any of):
  - Relative standing below 0.88 (squad projected pts vs opponent average)
  - `_lost_pts_gap > 50` (accumulated T1/T2 pts lost to opponents)
  - Pace signal: `my_cash_per_slot < opp_avg_cash_per_slot × 0.75`
- **Recovery mode**: bids 2.2× on the substitute target; suppresses off-plan players

## Player Data Model

```python
{
  "player_name":        "SP Narine",       # internal key (csv-style Cricsheet name)
  "auction_name":       "Sunil Narine",    # display name (from player_master.json)
  "role":               "AR",              # BAT | BOWL | AR | WK
  "tier":               1,                 # 1 = elite, 2 = solid, 3 = depth
  "projected_points":   91.2,
  "base_price":         2.0,
  "vorp":               35.1,              # value over replacement player
  "ipl_team":           "KKR",
  "auction_set":        3,
  "nationality":        "WI"
}
```

## Scoring Model

`Manager.season_score()` is C/VC-aware:
- Base: sum of all 16 players' `projected_points`
- Per match in `ipl_2025_schedule.json`: +1× best player pts (Captain), +0.5× second best (VC)
- Eligible C/VC players identified by `ipl_team` matching that match's teams
